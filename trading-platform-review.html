GOOD

Dual-stream ensemble design

Combining TFT (ML) with Lean rule-based signals via weighted ensemble is the right architecture. Reduces single-model failure risk significantly.

GAP

No feedback loop from execution

Execution results flow to monitoring but not back to the signal engine. PnL-aware signal weighting should be a Phase 2 priority. Without this, the ensemble weights (TFT=0.5 etc.) are static and never self-correct.

MISSING

Exit Monitor not wired to Kafka

The exit_monitor service exists but there's no Kafka topic for it in the README. TP/SL exits need a dedicated topic (e.g. exit.signals) flowing into execution — otherwise exits may race with entries.

GOOD

Kafka topic structure is clean

Clear producer/consumer separation. Topics follow a logical progression from raw data to approved risk. Good foundation for exactly-once delivery (Phase 5).

Business Logic Issues & Improvements

FIX

Static ensemble weights — biggest logic gap

Your weights (TFT=0.5, RSI=0.2, EMA=0.2, MACD=0.1) are hardcoded and never update. In different market regimes these weights will be wrong. At minimum, add a regime-conditional override now (don't wait for Phase 3).

# Current (brittle) score = 0.5*tft + 0.2*rsi + 0.2*ema + 0.1*macd # Better: regime-aware weights weights = { "bull": {"tft": 0.5, "rsi": 0.15, "ema": 0.25, "macd": 0.1}, "bear": {"tft": 0.4, "rsi": 0.3, "ema": 0.1, "macd": 0.2}, "sideways":{"tft": 0.3, "rsi": 0.25, "ema": 0.15, "macd": 0.3}, } w = weights[current_regime] score = w["tft"]*tft + w["rsi"]*rsi + w["ema"]*ema + w["macd"]*macd
FIX

Walk-forward gate only checks Sharpe — not drawdown

OOS Sharpe ≥ 0.5 and win rate ≥ 45% are necessary but not sufficient. A model can pass both with a catastrophic max drawdown. Add a drawdown constraint to the promotion gate.

Add to promotion gate: max_drawdown < 15%, Sortino ≥ 0.8, and Calmar ≥ 0.3. These catch strategies that win often but blow up on losses.
WARN

CVaR in Risk Engine — confidence level unspecified

The README mentions CVaR limits but doesn't specify the confidence level (95%? 99%?) or lookback window. This is critical — CVaR at 95% vs 99% gives very different position limits. Document and enforce this explicitly.

# Add to configs/app.yaml risk: cvar_confidence: 0.99 # 99% CVaR cvar_lookback_days: 252 # 1 year max_cvar_pct: 0.02 # 2% of portfolio per trade position_cap_pct: 0.10 # 10% max single position
WARN

Lean Alpha signals use fixed EMA periods

EMA-cross (12/26) is hardcoded — these are defaults from QuantConnect but not optimal for all assets/timeframes. Consider parameterizing in configs and running a grid search in backtester before promoting to live.

SUGGEST

Add signal confidence threshold — not just direction

Currently signals likely produce BUY/SELL/HOLD. Add a minimum confidence threshold before the signal is forwarded to the risk engine. Low-confidence signals should be dropped, not passed forward with a weak score.

MIN_CONFIDENCE = 0.62 # Only pass signals above this threshold if ensemble_confidence < MIN_CONFIDENCE: emit_to("risk.rejected", reason="low_confidence") return
GOOD

Model hot-reload is correctly designed

Green/blue model swap without restart is the right approach. Ensure the promotion script validates that the new model's input schema matches the current feature pipeline before swapping — a schema mismatch at inference is a silent killer.

Claude Code Prompt Improvements

📌 Problem: Your current prompts are too vague

Generic prompts like "fix the business logic" make Claude guess context. Good Claude Code prompts are specific, scoped, and include constraints.

1. Asking for a code review

❌ Before (weak)
"Check the business logic and make it amazing"
✅ After (strong)
"Review services/signal_engine/main.py. Focus on: (1) the ensemble weighting logic — are weights applied correctly across all signal types? (2) error handling if lean-alpha is down — does it gracefully degrade? (3) any race conditions in Kafka consumption. Don't touch the Kafka topic names."

2. Asking to fix a bug

❌ Before
"Fix the risk engine"
✅ After
"In services/risk_engine/risk.py, the CVaR calculation is returning NaN when there are fewer than 30 data points in the lookback window. Add a guard: if n_samples < 30, reject the signal with reason='insufficient_data'. Preserve the existing CVaR formula and don't change the position cap logic."

3. Adding a new feature

❌ Before
"Add regime detection"
✅ After
"Add a RegimeDetector class to shared/core/regime.py. It should classify market regime as 'bull', 'bear', or 'sideways' using 200-day SMA vs current price and rolling 20-day volatility percentile. Output: {regime: str, confidence: float 0-1}. Write unit tests in tests/unit/test_regime.py. Don't integrate into signal_engine yet — I'll wire it manually."

4. Asking for architecture advice

❌ Before
"How should I improve the platform?"
✅ After
"Given our current stack (FastAPI microservices, Kafka, TimescaleDB, ONNX TFT model), what's the highest-leverage improvement to signal quality for Phase 2? Consider: we have 12-month historical data, 9 services, paper trading only. Rank top 3 options by expected Sharpe improvement vs implementation effort. Be specific about what code would change."
TEMPLATE

Universal Claude Code prompt template for this project

Context: [service name, file path, what it currently does] Problem: [specific issue or goal — 1-2 sentences] Constraints: [what NOT to change, API contracts to preserve] Output: [what you want back — code, test, explanation, diff] Success: [how to verify it worked]
Use this template for every Claude Code request. It reduces back-and-forth by ~70% and prevents Claude from making unintended changes to other services.

Safety Gate Analysis

✓ GOOD

paper_trade=True hardcoded

Hardcoded in execution service. This is the right first layer — can't be disabled by a config typo.

✓ GOOD

File-based approval gate

.tmp/APPROVE_EXECUTION must exist. Good pattern — survives container restarts, easy to audit in logs.

⚠ WEAK

execution_dry_run in YAML

A YAML config flag as a safety gate is fragile — a misconfigured deploy or env var override can silently disable it. Move this check into code, not config.

Fix: Read this from an environment variable that defaults to true — not a YAML file that can be overwritten.
MISSING

No circuit breaker on kill switch

11 Prometheus alert rules are great, but what triggers automatic halt? If Kafka consumer lag spikes or model drift is detected, is there an automated kill switch, or just an alert that requires human action?

Add: Auto-revoke execution approval when drift_alert fires for >2 consecutive checks. Log the reason to TimescaleDB.
MISSING

No position-level reconciliation

There's no mention of a reconciliation step that checks Alpaca's actual open positions against what the risk engine thinks is open. In paper trading this is fine, but as you approach live, a missed execution or duplicate order would corrupt the risk model's state. Add a reconciliation job (cron every 15 min) to the orchestration service now while it's easy.
